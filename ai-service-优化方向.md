# AI-Service 模块优化方向

## 文档定位

本文档只记录 `ai-service` 的优化项，不重复讲全项目架构。

## 当前现状

`ai-service` 当前已经具备：

- 商品问答
- 会话管理
- 候选商品解析
- 知识库同步与统计
- Redis 会话缓存

近期已完成的正确性修复：

- 修复 AI 会话上下文缓存 miss 被误判为空上下文的问题

## P0：优先继续处理的项

### 1. 远程服务重复调用收敛

现状：

- 会话修复、商品身份补齐、问答路由过程中，仍可能出现重复商品查询

建议：

- 给商品身份补齐建立更明确的缓存策略
- 把会话修复和问答时的远程依赖读取收敛到单点

### 2. Embedding 与模型降级策略增强

现状：

- 模型不可用时已有降级，但策略仍偏粗

建议：

- 区分“模型超时”“模型限流”“向量不可用”“知识库为空”
- 为不同失败类型返回更稳定的 fallback

### 3. Prompt 与路由输出可观测性增强

建议：

- 显式记录命中的路由策略
- 记录 fallback 原因
- 对敏感信息做脱敏后输出关键调试字段

## P1：中期优化项

### 4. `RagRouteExecutionService` 职责拆分

现状：

- 路由执行逻辑承担了较多上下文组织、知识拼装和回答生成职责

建议：

- 拆成更清晰的 execution、prompt、retrieval、fallback 几层

### 5. 会话上下文模型继续增强

建议：

- 明确“当前商品”“候选比较商品”“最近解析结果”这几类状态的生命周期
- 减少上下文字段隐式耦合

### 6. 知识同步任务状态更完整

建议：

- 细化同步任务的阶段状态
- 增加失败原因和统计输出

## P2：低优先级优化项

### 7. Prompt 模板外部化

建议：

- 将部分长 Prompt 从代码中抽出，便于维护和版本管理

### 8. 命中率与延迟指标沉淀

建议：

- 统计命中率
- 统计 fallback 比例
- 统计各路由耗时

## 推荐推进顺序

1. 收敛重复远程调用
2. 增强模型降级策略
3. 增强日志与可观测性
4. 拆分路由执行职责
5. 优化会话上下文模型
