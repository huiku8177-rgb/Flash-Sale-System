# 当前阶段规划与后续路线

> 本文档用于说明“项目现在做到哪里了”以及“接下来最值得继续做什么”。

## 1. 当前项目阶段

项目已经从早期的秒杀接口练习，演进成一套可以联调演示的轻量商城系统。

当前有两条核心业务线：

- 普通商品链路
- 秒杀商品链路

这两条链路都已经具备基本闭环，但在商城化、运营化和工程化层面还有进一步完善空间。

## 2. 当前已经完成的能力

### 2.1 认证与账户

- 登录
- 注册
- 获取当前用户信息
- 修改密码
- 收货地址管理

### 2.2 普通商品链路

- 商品列表与详情
- 商品名搜索
- 分类筛选
- 分类关键词搜索命中
- 购物车增删改查
- 独立结算页
- 普通订单创建、查询、支付、取消

### 2.3 秒杀链路

- 秒杀商品列表与详情
- 发起秒杀
- 秒杀结果轮询
- Redis + Lua 控制库存
- RabbitMQ 异步建单
- 秒杀订单查询、支付、取消

### 2.4 前端页面

- 登录页
- 注册页
- 商城首页
- 秒杀页
- 购物车页
- 结算页
- 订单中心
- 账户信息页
- 密码安全页

### 2.5 工程能力

- Gateway 统一入口
- Swagger 聚合
- Nacos 注册中心
- Nacos 配置中心
- 基础限流与日志

## 3. 当前还值得继续推进的方向

## 3.1 让商城主链路更完整

优先级最高的业务增强仍然是普通商城体验：

- 结算预览接口
- 首页运营配置接口
- 统一订单聚合接口
- 首页统计概览接口

这部分补齐之后，前端页面的稳定性和真实感会明显提升。

## 3.2 增强后台与运营能力

目前项目偏向用户前台体验，后台管理能力几乎没有。

下一步适合考虑：

- 商品管理
- 秒杀活动配置
- 库存预热与重载
- 异常订单查询与补偿

## 3.3 提升工程质量

当前建议继续做的工程化工作：

- 单元测试与集成测试
- 更完整的链路日志与 `requestId`
- 关键指标监控
- 文档与配置持续同步

## 4. 推荐优先级

### P0：先把商城前台体验补稳

1. 结算预览接口
2. 首页运营配置接口
3. 统一订单聚合接口
4. 首页统计概览接口

### P1：补后台与运营入口

1. 商品管理接口
2. 秒杀活动配置接口
3. 运营位配置
4. 库存重载与异常补偿查询

### P2：继续补工程质量

1. 单元测试
2. 集成测试
3. 统一日志格式
4. 监控与告警

## 5. 推荐推进顺序

### 第一阶段：补前台最缺的接口

1. 结算预览
2. 首页运营配置
3. 统一订单聚合
4. 首页概览统计

### 第二阶段：补后台运营能力

1. 商品管理
2. 秒杀活动管理
3. 运营位配置

### 第三阶段：做工程化收尾

1. 测试补齐
2. 日志与监控
3. 配置与部署文档进一步收敛

## 6. 当前阶段的判断

如果现在只选一个方向继续做，我最推荐优先推进：

1. 结算预览接口
2. 首页运营配置接口

这两块做完后，用户视角下的“商城完成度”会提升最快。
