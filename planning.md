# 当前阶段计划与后续业务规划

> 本文件基于当前仓库真实实现整理，不再沿用旧的 `/api/**` 历史草案。  
> 目标是同时说明“现在已经完善到哪里”和“后面还要继续开发什么”。

## 1. 当前项目阶段

项目已经从早期的“秒杀后端原型”演进为“可联调的轻量商城系统”。

当前有两条核心业务线：

- 普通商品链路
- 秒杀商品链路

两条链路都已经具备基础闭环能力，但商城化业务仍有继续扩展空间。

## 4. 当前仍未完成的业务能力

下面这些不是“技术优化项”，而是后续真正值得继续做的业务功能。

### 4.1 购物车业务

当前购物车仍是前端本地状态，缺少真实后端能力：

- 购物车列表
- 加入购物车
- 修改数量
- 删除购物车商品
- 清空购物车
- 跨端同步购物车

### 4.2 用户地址业务

当前普通订单使用 `addressSnapshot` 手动传快照，缺少正式地址中心：

- 地址列表
- 新增地址
- 编辑地址
- 删除地址
- 默认地址
- 结算页选择地址

### 4.3 用户资料业务

当前用户信息较轻，后续可继续补：

- 昵称
- 头像
- 手机号
- 邮箱
- 编辑个人资料

### 4.4 统一订单中心业务

当前普通订单和秒杀订单由前端聚合展示，后续可升级为统一订单中心：

- 统一订单列表接口
- 订单分页
- 按状态筛选
- 按订单类型筛选
- 统一订单摘要字段

### 4.5 首页运营业务

当前首页还是前端拼装，缺少后端运营配置：

- 首页 banner 配置
- 首页楼层配置
- 频道导航配置
- 热门搜索词
- 搜索联想

### 4.6 秒杀运营业务

当前秒杀链路能跑，但运营感还不够强：

- 秒杀场次
- 按场次查询秒杀商品
- 排队位置
- 预估等待时间
- 活动维度展示

### 4.7 商品增强业务

当前商品浏览够用，但还不够像完整商城：

- 商品分类服务
- 搜索页
- 推荐商品
- 收藏商品
- 浏览历史

### 4.8 订单生命周期业务

当前支付是模拟支付，订单生命周期还可以继续补：

- 取消订单
- 超时未支付自动取消
- 发货
- 确认收货
- 完成订单
- 售后 / 退款状态

## 7. 前后端协作约定

- 新增业务接口时，必须同步更新 `前后端交互规范.md`
- 修改网关路由或鉴权规则时，必须同步更新 `README.md`
- 修改服务边界时，必须同步更新 `Flash-Sale-System架构与接口规范.md`
- 前端不再新增历史兼容入口，例如 `/pay-product/**`、`/seckill-order/**`
- 普通订单与秒杀订单允许后端继续分开建模，但前端展示尽量统一

## 8. 当前需要长期记住的限制

- 所有前端接口统一走 Gateway
- 除登录和注册外，默认都需要登录
- 商品接口当前也受鉴权保护
- 购物车当前仍是前端本地状态，不应误写成“已完成后端购物车能力”
- 模拟支付当前只用于闭环后端接口，不代表真实支付系统已完成

## 9. 下一步建议

如果按“业务价值 + 当前基础”来排，我建议下一步优先做：

1. 真实购物车服务
2. 地址管理
3. 统一订单中心接口

这三项做完之后，整个项目会从“可联调的练手商城”更进一步变成“结构完整的商城后端原型”。

目标：

主要业务功能：

- 真实购物车服务
- 地址管理
- 统一订单中心接口

涉及服务：

- `order-service`
- `auth-service`
- `gateway`

- `cart_item`
- `user_address`

可能新增的核心接口：

- `PUT /cart/items/{itemId}`

### 第二期：首页与搜索增强

### 第三期：秒杀运营化增强

## 11. 每期建议的数据库策略

### 第一期

- 先补 `cart_item`、`user_address`
- 优先保证表结构简单稳定，不急着做过度抽象

### 第二期

- 首页与搜索配置表可以按“运营配置”思路设计
- 先满足前端展示，不必一次做成复杂 CMS

### 第三期

- 秒杀活动与商品关系表建议独立，避免直接把活动概念硬塞进 `seckill_product`

### 第四期

- 订单状态流转建议先补状态字段与时间字段，再考虑售后子系统拆分

## 12. 每期建议的前后端分工

### 第一期

- 后端先补购物车、地址、统一订单接口
- 前端再把本地购物车切换到服务端购物车

### 第二期

- 后端先补首页配置与热词接口
- 前端再替换掉静态 banner、频道和热词

### 第三期

- 后端先补秒杀场次和活动商品接口
- 前端再重构秒杀页顶部结构和活动切换逻辑

### 第四期

- 后端先补订单状态流转接口
- 前端再细化订单操作按钮和状态展示

## 13. 推荐的项目推进方式

为了避免后面越做越乱，建议每一阶段都按下面顺序推进：

1. 先定数据模型
2. 再定接口契约
3. 然后写后端实现
4. 最后接前端页面
5. 阶段完成后同步更新文档

这样你后面每推进一期，都能保持“代码、接口、前端、文档”四者一致。

## 14. 第一期细化任务清单（迭代版）

> 本节是在现有计划基础上的进一步拆解，不替换前文，只作为第一期的执行清单。

### 14.1 第一期目标再确认

第一期聚焦三件事：

1. 真实购物车服务
2. 地址管理
3. 统一订单中心接口

这一期的核心目标不是“继续加页面”，而是把普通商城链路从“可演示”推进到“结构完整”。

### 14.2 第一期后端拆解

#### A. 购物车服务

建议优先放在 `order-service`，先不要为了拆服务而拆服务。

建议新增：

- `CartController`
- `CartService`
- `CartServiceImpl`
- `CartItemMapper`
- `CartItemMapper.xml`
- `CartItemPO`
- `CartItemVO`
- `AddCartItemDTO`
- `UpdateCartItemDTO`

建议接口：

- `GET /cart/items`
- `POST /cart/items`
- `PUT /cart/items/{itemId}`
- `DELETE /cart/items/{itemId}`
- `DELETE /cart/items`

建议规则：

- 普通商品和秒杀商品都允许进购物车，但字段里明确 `productType`
- 秒杀商品进购物车只保留草稿语义，不直接等同于下单
- 同用户、同商品、同类型重复加入时，直接累加数量
- 修改数量时要校验商品当前库存和状态

建议 SQL 表：

```sql
cart_item
id
user_id
product_id
product_type
quantity
selected
create_time
update_time
```

#### B. 地址管理

建议优先放在 `auth-service` 或独立 `user-service` 之前先落在 `auth-service`，因为当前用户能力都在这里。

建议新增：

- `AddressController`
- `AddressService`
- `AddressServiceImpl`
- `UserAddressMapper`
- `UserAddressMapper.xml`
- `UserAddressPO`
- `UserAddressVO`
- `SaveAddressDTO`

建议接口：

- `GET /user/addresses`
- `POST /user/addresses`
- `PUT /user/addresses/{id}`
- `DELETE /user/addresses/{id}`

建议规则：

- 每个用户可维护多条地址
- 支持 `isDefault`
- 新增默认地址时，其他默认地址自动取消默认
- 删除默认地址后不强制自动补默认，但前端要兜底提示

建议 SQL 表：

```sql
user_address
id
user_id
receiver
mobile
province
city
district
detail
is_default
create_time
update_time
```

#### C. 统一订单中心接口

建议仍放在 `order-service`。

建议新增：

- `AllOrderController` 或直接扩展现有 `OrderController`
- `AllOrderService`
- 统一订单 VO，例如 `OrderSummaryVO`

建议接口：

- `GET /order/all-orders`

建议查询参数：

- `pageNum`
- `pageSize`
- `status`
- `orderType`

建议返回字段：

- `id`
- `orderNo`
- `orderType`
- `status`
- `amount`
- `summary`
- `createTime`

建议实现方式：

- 第一版不要强行做数据库 union
- 先分别查普通订单和秒杀订单
- 在 service 层聚合、排序、分页

这样迭代成本更低，也更容易和你当前结构兼容。

### 14.3 第一期前端拆解

#### A. 购物车页改造

当前 `CartView.vue` 已有本地购物车结构，第一期只需要把数据源从本地切到后端：

- 页面初始化改为拉 `GET /cart/items`
- 加购按钮改为调 `POST /cart/items`
- 数量修改改为调 `PUT /cart/items/{itemId}`
- 删除改为调 `DELETE /cart/items/{itemId}`
- 清空购物车可补一个按钮调 `DELETE /cart/items`

前端改动文件预计包括：

- `flash-sale-ui/src/api/cart.js`
- `flash-sale-ui/src/composables/useMallApp.js`
- `flash-sale-ui/src/views/CartView.vue`
- `flash-sale-ui/src/views/HomeView.vue`
- `flash-sale-ui/src/views/FlashSaleView.vue`
- `flash-sale-ui/src/views/AppShellView.vue`

#### B. 结算页改造

当前结算还是手填地址快照。第一期接地址管理后，可以改成：

- 默认读取地址列表
- 支持选择地址
- 下单时由前端把选中地址转成 `addressSnapshot`

这样能保留你现有普通订单表结构，不需要马上改订单表。

#### C. 个人中心改造

当前个人中心是前端聚合普通单和秒杀单。第一期接统一订单中心后，可以：

- 优先用 `/order/all-orders`
- 保留原有普通单、秒杀单接口作为详情和兼容接口

### 14.5 第一期验收清单

当下面这些都满足时，可以认为第一期完成：

#### 后端验收

- 购物车接口可增删改查
- 地址接口可增删改查
- 统一订单中心接口可按类型和状态查询
- 普通下单时可以复用地址数据形成快照

#### 前端验收

- 刷新页面后购物车数据不丢
- 用户可以直接选择地址下单

#### 文档验收

- README 已更新

### 14.6 第一期建议执行顺序

如果真的开始做，建议按这个顺序推进：

1. 先建 `cart_item`、`user_address`
2. 再补购物车接口
3. 再补地址接口
4. 再补统一订单中心接口
5. 最后再接前端页面

这样可以最大限度减少返工。

### 14.7 第一期之后紧接着可做什么

第一期结束后，最自然进入第二期的就是：

- 首页装修配置
- 热门搜索与搜索建议
- 商品分类服务

因为到那时商城“交易基础”已经稳了，再做首页和搜索，收益会更高。
