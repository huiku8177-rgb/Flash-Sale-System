# 当前阶段规划与后续路线

## 文档定位

本文档只回答一件事：项目下一阶段最值得做什么，以及为什么先做它。

它不重复描述启动步骤、接口细节和技术栈，分别交给其他文档维护。

## 当前阶段判断

项目已经不是单一的“秒杀接口练习”，而是一套可联调演示的轻量商城系统。

当前已形成 3 条主线：

- 普通商品链路
- 秒杀链路
- AI 助手链路

现阶段最明显的短板，不是基础能力缺失，而是“商城完整度”和“运营后台能力”还不够。

## 已完成的核心能力

### 账号与用户

- 登录、注册、退出登录
- 用户信息获取
- 密码修改
- 地址管理

### 普通商城

- 商品列表与详情
- 搜索与分类筛选
- 购物车
- 结算下单
- 订单查询、支付、取消

### 秒杀

- 秒杀商品列表与详情
- 秒杀入口
- 结果轮询
- Redis + Lua 预扣库存
- RabbitMQ 异步建单
- 秒杀订单查询、支付、取消

### AI

- 商品问答
- 会话管理
- 候选商品解析
- 知识库同步与统计

## 下一阶段优先级

### P0：先补商城闭环完整度

优先做：

1. 结算预览接口
2. 首页运营配置接口
3. 统一订单聚合接口
4. 首页统计与用户概览接口

原因：

- 这几项最直接提升前端页面的真实感
- 对现有链路改动相对小
- 能明显降低前端临时拼装逻辑

### P1：补后台与运营能力

建议推进：

1. 商品管理接口
2. 秒杀活动管理接口
3. 运营位配置
4. 库存预热与重载入口
5. 异常订单查询与补偿入口

原因：

- 当前系统更像“用户前台演示版”
- 缺后台能力会限制后续扩展和测试

### P2：继续补工程质量

建议推进：

1. 单元测试与集成测试补齐
2. 统一日志格式和 `requestId`
3. 指标监控与告警
4. 部署和联调脚本进一步收敛

## 推荐推进顺序

### 第一阶段

- 结算预览
- 首页运营配置
- 统一订单聚合
- 首页统计概览

### 第二阶段

- 后台商品管理
- 秒杀活动配置
- 运营位管理

### 第三阶段

- 测试补齐
- 日志与监控
- 部署文档与脚本收口

## 当前不建议优先做的事

以下方向可以做，但不建议先做：

- 过早引入复杂支付网关
- 大规模重构微服务边界
- 先做后台 UI 再补后端接口
- 先做性能压测平台再补业务闭环

原因很简单：这些都不是当前最短板。
