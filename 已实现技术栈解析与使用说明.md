# Flash Sale System：已实现技术栈解析与使用说明

本文聚焦当前仓库中**已经真正落地**的技术栈，以及在项目中的具体使用方式。

---

## 1. Java 17

### 为什么用

- 与 Spring Boot 3.x 版本匹配良好。
- 支持长期维护（LTS），利于项目迭代。

### 你是怎么用的

- 父工程统一定义 Java 版本，保证各服务编译环境一致。
- 微服务模块统一基于 Java 17 运行。

---

## 2. Spring Boot（多服务）

### 为什么用

- 快速启动、自动配置、适合业务拆分。
- 对 Web、配置、依赖管理生态完善。

### 你是怎么用的

- 每个服务都以独立 `SpringBootApplication` 启动：`gateway`、`auth-service`、`seckill-service`、`order-service`。
- 通过 Maven 多模块组织，共享 `common` 模块减少重复代码。

---

## 3. Spring Cloud Gateway

### 为什么用

- 作为统一流量入口，天然适合做路由和鉴权前置。
- 可以把安全、路由、跨服务请求头透传等横切逻辑集中管理。

### 你是怎么用的

- 在 `application.yml` 里配置了三条核心路由：`/auth/**`、`/order/**`、`/seckill/**`。
- 实现了 `GlobalFilter`：
  - 白名单路径直接放行；
  - 非白名单读取 Bearer Token；
  - 解析 JWT 后注入 `X-User-Id`。

---

## 4. JWT + RSA

### 为什么用

- 适合无状态认证，便于服务扩展。
- RSA 非对称签名更利于分布式场景下的验签安全性。

### 你是怎么用的

- 将 JWT 工具抽到 `common` 模块，统一封装：
  - `createToken(userId, ttl)`：生成 token；
  - `parseToken(token)`：验签、验过期、取用户。
- 网关使用同一工具完成统一鉴权。
- `auth-service` 登录成功后签发 token 返回前端。

---

## 5. Spring Security Crypto（BCrypt）

### 为什么用

- 明文密码不可接受，BCrypt 是成熟方案。

### 你是怎么用的

- 注册接口：对密码做 `passwordEncoder.encode()` 后入库。
- 登录接口：通过 `passwordEncoder.matches()` 比对密码。

---

## 6. MyBatis（注解方式）

### 为什么用

- 登录注册场景 SQL 简单，注解方式开发效率高。

### 你是怎么用的

- `UserMapper` 里通过 `@Select`/`@Insert` 完成用户查询与新增。
- `auth-service` 的业务层直接调用 Mapper 完成认证流程。

---

## 7. Nacos 服务发现

### 为什么用

- 微服务之间通过服务名访问，避免写死地址。

### 你是怎么用的

- 各模块配置统一的 Nacos 地址与 namespace。
- Gateway 路由使用 `lb://service-name` 进行负载与发现。

---

## 8. Maven 多模块工程

### 为什么用

- 清晰分层：网关、业务服务、公共库解耦。
- 统一依赖与版本，降低维护成本。

### 你是怎么用的

- 父 POM 统一管理 Spring Boot / Spring Cloud / Hutool / MyBatis 版本。
- 子模块按职责引入依赖，公共能力（Result、JwtTool、异常）沉淀到 `common`。

---

## 9. 当前技术栈落地状态（客观）

### 已经实装并可说明的

- Java 17
- Spring Boot 微服务基础
- Spring Cloud Gateway 路由与鉴权
- JWT + RSA 签发/验签
- BCrypt 密码加密
- MyBatis 注解式 DAO
- Nacos 服务发现
- Maven 多模块组织

### 已配置但业务能力尚未完整落地

- Redis（秒杀库存、Lua 原子扣减）
- RabbitMQ（异步下单消费者链路）
- 秒杀核心领域模型与订单领域模型

---

## 10. 你这套实现的架构价值

- 先把“认证与网关入口”打通，形成可扩展的安全底座。
- 把横切能力（JWT、统一响应）沉淀在公共模块，后续业务模块接入成本低。
- 为秒杀核心能力（Redis/Lua/MQ）留好了清晰接入点。

如果下一阶段继续推进，建议按“商品查询 -> 秒杀接口 -> Redis Lua -> MQ -> 订单查询”的顺序迭代，保持每一步都可联调可验收。
