# 前端完整化待补充后端接口清单

> 本文档只记录“当前仍值得继续补齐”的后端能力。  
> 已经完成并联调通过的能力，不再重复写入本清单。

## 1. 已经完成的主链路

当前前端已经可以正常联调这些能力：

- 登录、注册、退出登录
- 获取当前用户信息
- 修改密码
- 收货地址增删改查与默认地址
- 普通商品列表、详情、搜索、分类筛选
- 秒杀商品列表与详情
- 购物车增删改查
- 独立结算页
- 基于购物车创建普通订单
- 普通订单列表、详情、模拟支付、取消、支付状态查询
- 发起秒杀、轮询秒杀结果
- 秒杀订单列表、详情、模拟支付、取消、支付状态查询

## 2. 当前仍值得继续补的后端能力

### 2.1 结算预览接口

当前问题：

- 结算页虽然能正常下单，但前端仍需要自己组合较多展示状态
- 缺少统一的“本次可结算结果”接口

建议新增接口：

- `GET /product/cart/preview`
  或
- `POST /product/cart/preview`

建议返回内容：

- 已选中的可结算商品
- 失效商品
- 库存不足商品
- 合计金额
- 可结算件数
- 地址校验结果

收益：

- 结算页展示更稳定
- 后端成为金额和可支付状态的唯一口径
- 更适合后续扩展优惠、运费和活动规则

### 2.2 首页运营配置接口

当前问题：

- 首页热词、频道、轮播、推荐位仍偏前端静态配置

建议新增接口：

- `GET /home/config`
- `GET /home/channels`
- `GET /search/hot-keywords`

建议返回内容：

- 首页轮播卡片
- 热门搜索词
- 首页频道导航
- 首页推荐商品分组

收益：

- 首页更像真实商城
- 运营位调整不需要前端改代码

### 2.3 统一订单聚合接口

当前问题：

- 前端订单中心仍需分别查询普通订单与秒杀订单，再在前端聚合

建议新增接口：

- `GET /order/all-orders`

建议支持参数：

- `status`
- `orderType`
- `pageNum`
- `pageSize`

收益：

- 订单中心数据装配更轻
- 过滤、分页和排序语义更统一

### 2.4 首页统计与用户概览接口

当前问题：

- 首页右侧最近订单、统计卡片目前主要靠多个接口组合

建议新增接口：

- `GET /dashboard/summary`

建议返回内容：

- 普通商品总数
- 秒杀商品总数
- 待支付订单数
- 最近订单列表

收益：

- 首页和个人中心首屏可以更稳定
- 降低前端多接口拼装成本

### 2.5 后台商品与秒杀运营接口

当前问题：

- 当前仓库偏向商城前台体验
- 缺少真正的后台运营入口

建议新增接口：

- 商品上架 / 下架
- 商品新增 / 编辑
- 秒杀商品预热
- 秒杀活动配置
- 库存重载与补偿查询

收益：

- 项目更像完整系统，而不只是演示前台

### 2.6 真实支付回调或更完整的支付态流转

当前问题：

- 现在是模拟支付
- 订单状态已经有基础流转，但缺少真正支付接入语义

建议方向：

- 支付创建接口
- 支付回调接口
- 更完整的支付中、支付失败、退款等状态扩展

收益：

- 更贴近真实订单生命周期

## 3. 推荐推进顺序

建议按下面顺序继续补：

1. 结算预览接口
2. 首页运营配置接口
3. 统一订单聚合接口
4. 首页统计与用户概览接口
5. 后台商品与秒杀运营接口
6. 真实支付链路
